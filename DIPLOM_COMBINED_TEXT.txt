2.2 Практическая часть

2.2.1 Разработка базы данных

Для хранения пользовательской информации, игровых данных и сетевого состояния матчей в проекте была разработана реляционная база данных на основе SQL Server. Использование СУБД позволило обеспечить централизованное хранение информации, поддержку сетевого режима и синхронизацию данных между клиентом и сервером. В процессе проектирования были определены основные сущности системы, а также связи между ними. База данных обеспечивает хранение учетных записей пользователей, игровых профилей, коллекций карт, статистики матчей, информации о лобби и состоянии сетевых игр.

[ВСТАВИТЬ: ER-диаграмму базы данных]

Основными таблицами базы данных являются players, auth_users, player_statistics, player_owned_cards, player_selected_deck, cards, public_lobbies, lobby_players, match_state, match_players и match_commands.

Таблица auth_users используется для хранения данных авторизации пользователей. В таблице содержатся логин, hash пароля и salt, используемый для повышения безопасности хранения учетных данных.

[ВСТАВИТЬ: скрин структуры таблицы auth_users]

Таблица players хранит основную информацию о профиле игрока, включая никнейм, уровень, количество монет и рейтинг. Таблица player_statistics содержит статистические данные, в том числе количество матчей, число побед, число поражений, а также производные показатели, характеризующие успешность игрока.

[ВСТАВИТЬ: скрин SQL Server с таблицами]

Для хранения коллекции карт используется таблица player_owned_cards. Она содержит информацию о картах, принадлежащих пользователю, а также сведения об источнике получения этих карт. Таблица player_selected_deck хранит выбранную игроком колоду и порядок расположения карт в ней. Таблица cards используется для хранения карточной базы и служит связующим элементом между игровой логикой и пользовательской коллекцией.

Отдельное внимание было уделено сетевому режиму. Для поддержки многопользовательского взаимодействия были созданы таблицы public_lobbies и lobby_players. Они обеспечивают хранение информации о созданных комнатах, хосте комнаты, максимальном числе участников, текущем статусе лобби и подключенных игроках.

Таблица match_state используется для хранения общего состояния игрового матча, а match_commands предназначена для записи действий игроков и передачи игровых команд между участниками. Дополнительно таблица match_players хранит состояние каждого участника матча, включая его идентификатор, позицию, карты в руке, очки влияния, политические очки и другие параметры, необходимые для синхронизации игрового процесса.

[ВСТАВИТЬ: пример SQL-запроса создания таблицы]

В процессе разработки были реализованы механизмы автоматического создания таблиц, обновления состояния матча, удаления завершённых лобби, очистки зависимых записей и синхронизации игровых данных. Использование SQL Server позволило обеспечить надежное хранение данных, повысить устойчивость системы к потере информации и подготовить основу для расширения сетевого режима.

2.2.2 Разработка программного продукта

Разработка программного продукта выполнялась в соответствии с ранее определёнными требованиями к системе. Основной целью являлось создание функциональной карточной игры с поддержкой одиночного и сетевого режимов, системой профиля игрока, хранением пользовательских данных и современной архитектурой клиент серверного взаимодействия.

В качестве основного инструмента разработки использовался игровой движок Unity. Выбор данного движка обусловлен широкими возможностями по созданию интерактивных приложений, развитой системой пользовательского интерфейса, поддержкой языка программирования C#, а также наличием большого количества встроенных инструментов для работы с графикой, сценами, анимацией и сетевым взаимодействием. В ходе работы была реализована клиентская часть игры, игровая логика, система интерфейсов, а также серверный backend для сетевого взаимодействия.

[ВСТАВИТЬ: скрин проекта в Unity Editor]

В процессе разработки была сформирована структура проекта, включающая клиентскую часть, backend API, систему хранения данных и отдельные сервисы для сетевого взаимодействия. Архитектура проекта построена по модульному принципу, что позволяет упростить сопровождение проекта и обеспечить возможность дальнейшего расширения функциональности. Такой подход позволил уменьшить связанность компонентов и сделать систему пригодной для дальнейшего масштабирования.

В структуре проекта были выделены следующие основные подсистемы: система игровой логики, система отображения интерфейса, система хранения данных, система авторизации, система работы с карточками, система сетевого взаимодействия, система сохранения профиля и система управления сценами.

[ВСТАВИТЬ: диаграмму архитектуры проекта]

Основой клиентской части является набор сцен, каждая из которых отвечает за определенный этап работы пользователя. В проекте были подготовлены главное меню, одиночный режим, сетевой режим и сцена тестирования. Для автоматизации инициализации сцен были реализованы bootstrap компоненты. Они выполняют автоматическую настройку интерфейса, загрузку данных профиля, повторную привязку игровых сервисов и инициализацию сетевых компонентов. В проекте данную роль выполняют MainMenuSceneBootstrap, LanGameSceneBootstrap и PersistentRuntimeServices.

[ВСТАВИТЬ: скрин MainMenuScene в Unity]

Для организации игровой логики был реализован центральный менеджер матча GameManager. Данный компонент отвечает за обработку игрового процесса, контроль раундов, определение текущего игрока, запуск голосований и обработку игровых событий. В процессе разработки была реализована полноценная пошаговая логика, включающая систему ходов, систему очков действия, систему влияния, механизм голосований, обработку игровых эффектов и проверку условий завершения матча.

[ВСТАВИТЬ: код GameManager]

Для хранения состояния партии был разработан класс GameState. Он содержит информацию о текущем раунде, активном игроке, состоянии участников, доступных картах и игровых событиях. Каждый участник матча представлен отдельным объектом ParticipantState. В нём хранятся количество очков влияния, количество политических очков, карты в руке, текущее состояние игрока, активные эффекты и временные модификаторы. Использование отдельных классов состояния позволило упростить обработку игровой логики и повысить читаемость программного кода.

Дополнительно была реализована система ActionLog, обеспечивающая запись всех игровых действий в журнал событий. Это позволяет пользователю отслеживать происходящие события во время матча.

[ВСТАВИТЬ: скрин игрового лога]

Одной из ключевых частей проекта является система карт. В игре реализована data driven архитектура, при которой база карт хранится в JSON файлах и загружается динамически. Для работы с картами были разработаны CardDefinition, CardDatabase, JsonDatabase и CardArtLibrary. Каждая карта содержит идентификатор, название, описание, стоимость, тип, редкость и игровой эффект. Дополнительно каждая карта может содержать числовые параметры, длительность и вероятности срабатывания в зависимости от типа эффекта.

[ВСТАВИТЬ: пример cards.json]

Важной частью разработки стала система карт. Для хранения карточной базы используются JSON файлы cards.json и events.json. Загрузка данных выполняется автоматически при запуске игры через компонент JsonDatabase. Такой подход обеспечивает возможность добавления новых карт без необходимости изменения программного кода. Для визуального отображения карточек используется библиотека CardArtLibrary, обеспечивающая подгрузку изображений и управление артами карт.

[ВСТАВИТЬ: скрин карточки в игре]

В ходе разработки были реализованы различные типы карт, включая карты влияния, карты событий, временные эффекты, карты усиления и специальные политические действия. Для обработки эффектов используется компонент CardResolver. Он отвечает за применение игровых изменений к состоянию участников. Система эффектов поддерживает изменение очков влияния, блокировки, временные штрафы, усиления, изменение политических очков и воздействие на других участников матча.

[ВСТАВИТЬ: код CardResolver]

Отдельное внимание было уделено разработке пользовательского интерфейса. Интерфейс игры выполнен в стилистике dark fantasy с использованием декоративных элементов, плашек, рамок и тематических текстур. При разработке интерфейса использовались TextMeshPro, UI Canvas, layout groups, динамическая генерация элементов и система автоматического масштабирования.

[ВСТАВИТЬ: скрин интерфейса]

В процессе работы была реализована система генерации интерфейса из программного кода. Такой подход позволил автоматически создавать панели, кнопки, списки, окна, элементы карточек и панели статистики. Данный механизм существенно упростил поддержку старых сцен и позволил избежать необходимости полной переработки интерфейса после внесения изменений.

Особое место заняла работа с интерфейсным атласом и UI плашками. Были исправлены ошибки автоматической нарезки атласа, убраны лишние slices, приведены в порядок реальные элементы спрайтового набора, настроены координаты, margins и корректное 9 slice поведение. Благодаря этому удалось уменьшить деформации углов, стабилизировать растяжение панелей и повторно использовать декоративные элементы во всех частях интерфейса.

Bootstrap система выполняет загрузку профиля, создание недостающих UI элементов, повторную привязку сервисов, инициализацию сетевых компонентов и настройку контроллеров интерфейса. Важным результатом стала возможность точечной достройки старых сцен без полного rebuild интерфейса, что позволило динамически добавлять TutorialPanel, TutorialButton, RewardsScroll и другие недостающие элементы.

[ВСТАВИТЬ: код bootstrap инициализации]

Важной частью проекта является система профиля игрока. В ходе разработки была реализована возможность регистрации, авторизации, хранения статистики, сохранения коллекции карт и синхронизации прогресса. Для реализации авторизации был создан backend API на ASP.NET Core. Серверная часть обеспечивает регистрацию пользователей, вход в систему, генерацию JWT токенов, проверку авторизации, хранение данных профиля и управление сетевыми лобби.

[ВСТАВИТЬ: скрин backend API или Swagger]

В процессе разработки были созданы следующие API маршруты: /api/auth/register, /api/auth/login, /api/profile/me, /api/lobbies и /api/online. Дополнительно были реализованы маршруты для обновления колоды, изменения числа монет, открытия новых карт, фиксации результатов матча и сохранения сетевого состояния.

Для хранения данных используется SQL Server. База данных обеспечивает хранение аккаунтов, статистики, коллекции карт, сетевых матчей и игровых команд. Дополнительно была реализована система кэширования профиля и fallback механизм. В случае недоступности сервера система может использовать локальное JSON хранилище.

[ВСТАВИТЬ: схема БД]

Одной из наиболее сложных частей проекта стала реализация сетевого взаимодействия. В проекте подготовлены одиночный режим, LAN взаимодействие и серверная онлайн модель. Для сетевого режима были разработаны OnlineLobbyManager, LanMatchCoordinator, NetworkGameManager и LobbyUIController.

[ВСТАВИТЬ: скрин лобби]

Сетевой режим поддерживает создание комнаты, подключение игроков, ready state, запуск матча, синхронизацию состояния и обмен игровыми командами. Для хранения сетевого состояния были реализованы специальные таблицы public_lobbies, lobby_players, match_state, match_players и match_commands. Таблица public_lobbies хранит сведения о комнатах, lobby_players хранит участников лобби, match_state содержит общий snapshot матча, match_players хранит состояние каждого игрока, а match_commands служит для записи действий участников и передачи игровых команд между клиентами через серверную часть.

В ходе разработки была реализована система snapshot состояния матча. Она позволяет сохранять текущее состояние игры и синхронизировать его между участниками. В snapshot включены данные о текущем раунде, очередности хода, очках действия, влиянии, политических очках, картах на руках игроков, результате партии и других параметрах.

[ВСТАВИТЬ: пример snapshot JSON]

Для матчей уже существовали структуры LanGameCommand и LanGameSnapshot, описывающие команды и состояние сетевого матча. Эти форматы были использованы как основа при развитии серверной модели, что позволило не переписывать всю боевую логику с нуля. Хостовая сторона формирует snapshot через GameManager, а клиентская часть получает актуальное состояние матча и применяет его к локальному интерфейсу. Одновременно backend сохраняет это состояние в SQL Server и раскладывает по match_state и match_players.

Дополнительно была проведена работа по оптимизации производительности системы. Были выполнены уменьшение количества rebuild интерфейса, оптимизация загрузки сцен, сокращение сетевых запросов, оптимизация layout системы и улучшение повторной инициализации сервисов.

Важное внимание уделялось исправлению визуальных проблем интерфейса. В процессе полировки были устранены вылезание элементов, наложение текста, деформация UI плашек, проблемы масштабирования, ошибки отображения карточек и проблемы preview в коллекции.

[ВСТАВИТЬ: сравнение до после UI]

Также была улучшена читаемость текста. В интерфейсе был повышен контраст, усилен font weight, переработаны отступы и унифицированы размеры элементов. Дополнительно были исправлены профиль игрока, экран коллекции, интерфейс магазина и визуальное оформление карточек. Были устранены серые заглушки у наборов карт, возвращена корректная подгрузка артов, улучшено окно наград и исправлено вылезание изображений за рамки карточек.

Таким образом, в ходе разработки был создан полноценный программный продукт, включающий клиентскую часть, backend API, базу данных, систему профиля, сетевое взаимодействие и современный пользовательский интерфейс. Разработанная архитектура обеспечивает возможность дальнейшего развития проекта, расширения функциональности и добавления новых игровых механик.

2.2.3 Описание программного продукта

Разработанный программный продукт представляет собой карточную политическую игру с поддержкой одиночного и сетевого режимов. После запуска приложения пользователь попадает в главное меню системы.

[ВСТАВИТЬ: скрин главного меню]

Главное меню включает кнопку начала игры, профиль пользователя, коллекцию карт, магазин, настройки, обучение и выход из приложения. Для авторизации пользователя реализована отдельная система входа и регистрации.

[ВСТАВИТЬ: окно авторизации]

После успешного входа выполняется загрузка профиля игрока и восстановление сессии через JWT токен. В системе реализован экран профиля, отображающий уровень игрока, статистику матчей, количество монет и число открытых карт. В профиле информация представлена в виде отдельных блоков статистики, что повышает наглядность данных и улучшает визуальное восприятие.

[ВСТАВИТЬ: скрин профиля]

Важной частью проекта является коллекция карт. Пользователь может просматривать доступные карты, фильтровать их по редкости и типу, а также формировать собственную колоду. В коллекции поддерживается preview карты, просмотр характеристик и изменение состава выбранной колоды.

[ВСТАВИТЬ: экран коллекции]

Внутриигровой магазин позволяет приобретать карточные наборы за игровую валюту.

[ВСТАВИТЬ: магазин наборов]

После покупки набора отображается окно наград с полученными картами. Вместо простого текста награды показываются в виде реальных карточек, что делает взаимодействие с магазином более наглядным.

Игровой процесс реализован в пошаговом режиме. Во время матча пользователь может разыгрывать карты, завершать ход, участвовать в голосованиях, применять эффекты и управлять ресурсами.

[ВСТАВИТЬ: скрин игрового HUD]

В верхней части интерфейса отображается информация о раунде и текущем ходе, а в центральной части показываются игровые события и используемые карты. Рука игрока автоматически перераскладывается, а игровые панели отображают влияние, политические очки и текущее состояние участников.

Для одиночного режима реализован искусственный интеллект BotAI, способный анализировать состояние партии и автоматически выполнять действия.

[ВСТАВИТЬ: скрин PvE матча]

В проекте также реализован сетевой режим с поддержкой онлайн лобби. Пользователь может создать комнату, подключиться к существующему лобби, выбрать готовность и начать матч.

[ВСТАВИТЬ: скрин лобби]

Сетевое взаимодействие выполняется через backend API и SQL Server. На текущем этапе реализовано хранение информации о комнатах, участниках, состоянии матча, командах игроков и snapshot данных игровой партии.

2.2.4 Разработка инструкции пользователя

Для обеспечения удобства использования программного продукта была разработана инструкция пользователя. После запуска приложения пользователю необходимо пройти процедуру авторизации или регистрации.

[ВСТАВИТЬ: окно регистрации]

После входа становится доступно главное меню игры. Для начала одиночной игры необходимо выбрать пункт PvE. После этого система автоматически создаёт матч против искусственного интеллекта.

Для участия в сетевой игре необходимо открыть раздел лобби, создать комнату либо подключиться к существующей.

[ВСТАВИТЬ: создание комнаты]

В разделе коллекции пользователь может просматривать карты, редактировать колоду и изучать характеристики карт. Для покупки новых наборов используется магазин игры.

В разделе настроек доступны изменение разрешения, регулировка громкости, выбор качества графики и переключение полноэкранного режима.

Таким образом, разработанная инструкция позволяет пользователю быстро освоить основные функции программного продукта и эффективно взаимодействовать с системой.

2.3 Мероприятия по охране труда и противопожарной безопасности

2.3.1 Описание требований по технике безопасности при работе с вычислительной техникой

Работа программиста непосредственно связана с длительным использованием вычислительной техники, периферийного оборудования и программных средств. В связи с этим особое внимание должно уделяться соблюдению требований техники безопасности, направленных на снижение вероятности возникновения травм, аварийных ситуаций и негативного воздействия на здоровье человека.

Рабочее место программиста должно соответствовать требованиям безопасности, установленным нормативными документами и стандартами в области эксплуатации вычислительной техники. Используемое оборудование должно находиться в исправном состоянии и иметь соответствующее заземление.

[ВСТАВИТЬ: фотографию рабочего места программиста]

Перед началом работы необходимо выполнить визуальную проверку оборудования. Следует проверить целостность кабелей, убедиться в отсутствии повреждений корпуса, проверить исправность сетевых подключений и убедиться в отсутствии посторонних предметов возле оборудования.

Не допускается эксплуатация техники при повреждении изоляции проводов, при наличии запаха гари, при перегреве оборудования, при нестабильной работе блока питания и при обнаружении механических повреждений.

Во время работы запрещается самостоятельно вскрывать оборудование, касаться внутренних компонентов подключенного устройства, использовать неисправные удлинители и перегружать электрическую сеть большим количеством устройств.

Для обеспечения безопасности необходимо соблюдать требования электробезопасности. Все устройства должны подключаться через исправные сетевые фильтры и источники бесперебойного питания.

Особое внимание следует уделять организации кабельного подключения. Провода не должны располагаться в проходах или создавать опасность случайного повреждения.

[ВСТАВИТЬ: схему безопасного подключения оборудования]

При возникновении аварийной ситуации необходимо отключить оборудование от сети, сообщить ответственному лицу, прекратить эксплуатацию неисправной техники и не пытаться самостоятельно устранить серьезные неисправности.

Дополнительно следует соблюдать правила пожарной безопасности. В помещении должны находиться исправные средства пожаротушения, а сотрудники обязаны знать порядок действий при возникновении возгорания.

2.3.2 Характеристика условий труда программиста

Работа программиста относится к категории умственного труда и сопровождается высокой нагрузкой на зрение, нервную систему и опорно двигательный аппарат. Основная часть рабочего времени связана с написанием программного кода, анализом данных, тестированием программного продукта, работой с документацией и взаимодействием с интерфейсами разработки.

[ВСТАВИТЬ: скрин среды разработки Visual Studio]

При длительной работе за компьютером могут возникать зрительное напряжение, утомляемость, снижение концентрации внимания, боли в спине и шее, а также перенапряжение кистей рук. Для снижения негативного воздействия необходимо правильно организовать рабочий процесс и соблюдать режим труда и отдыха. Рекомендуется выполнять короткие перерывы каждые 45–60 минут, использовать упражнения для глаз, изменять положение тела во время работы и поддерживать правильную осанку.

Большое значение имеет микроклимат помещения. Температура воздуха должна находиться в пределах 20–24 градусов, а влажность должна составлять 40–60 процентов.

Освещение рабочего места должно быть равномерным и достаточным для комфортной работы с текстовой информацией. Не допускается наличие резких теней, чрезмерной яркости и бликов на поверхности монитора.

Дополнительно следует учитывать уровень шума, создаваемого оборудованием. Шум не должен отвлекать работника и вызывать дополнительную нагрузку на нервную систему.

[ВСТАВИТЬ: схему правильного освещения рабочего места]

Правильная организация условий труда позволяет снизить утомляемость и повысить эффективность работы программиста.

2.3.3 Электромагнитное и ионизирующее излучения

Современная вычислительная техника является источником электромагнитного излучения. Несмотря на то, что современные устройства соответствуют требованиям безопасности, длительное воздействие электромагнитных полей может оказывать влияние на организм человека.

Основными источниками электромагнитного излучения являются мониторы, системные блоки, блоки питания, беспроводные устройства и сетевое оборудование.

[ВСТАВИТЬ: изображение рабочего компьютера]

Наибольшее внимание уделяется воздействию мониторов, так как пользователь находится в непосредственной близости к экрану в течение длительного времени. Для снижения уровня воздействия рекомендуется использовать современные LCD или LED мониторы, соблюдать расстояние 60–70 см до экрана, уменьшать время непрерывной работы и использовать оборудование с сертификацией безопасности.

Дополнительно необходимо правильно размещать оборудование в помещении. Мониторы не должны располагаться вплотную друг к другу, а системные блоки должны иметь достаточную вентиляцию. Современные компьютеры практически не создают опасного уровня ионизирующего излучения, однако соблюдение требований эксплуатации остаётся обязательным.

Также важным фактором является уровень статического электричества. Для его уменьшения необходимо поддерживать оптимальную влажность воздуха, использовать качественные материалы покрытия и регулярно проводить очистку оборудования.

Соблюдение перечисленных требований позволяет минимизировать негативное влияние электромагнитного излучения на пользователя.

2.3.4 Эргономические требования к рабочему месту

Эргономика рабочего места оказывает значительное влияние на комфорт работы и состояние здоровья программиста. Рабочее место должно быть организовано таким образом, чтобы минимизировать нагрузку на позвоночник, зрение и мышцы рук.

[ВСТАВИТЬ: схему эргономичного рабочего места]

Монитор должен располагаться на расстоянии 60–70 см от глаз, немного ниже уровня глаз и под небольшим углом наклона. Клавиатура и мышь должны располагаться таким образом, чтобы руки находились в естественном положении без чрезмерного напряжения.

Стул должен обеспечивать поддержку спины, регулировку высоты, удобное положение рук и устойчивость конструкции. Рабочий стол должен обладать достаточной площадью для размещения оборудования и документов.

Освещение рабочего места должно быть комбинированным и включать естественное и искусственное освещение. При этом необходимо избегать попадания прямого света на экран монитора.

В процессе работы рекомендуется изменять положение тела, выполнять разминку, делать перерывы и избегать длительного нахождения в одной позе.

Правильная организация рабочего места способствует снижению утомляемости, повышению производительности труда и уменьшению вероятности возникновения профессиональных заболеваний.

Основной целью разработки являлось создание функционального программного продукта, сочетающего в себе элементы карточной стратегии, системы развития профиля игрока и многопользовательского взаимодействия.

В процессе выполнения работы были изучены существующие аналоги и проведён анализ современных карточных игр. Это позволило определить основные требования к разрабатываемой системе и сформировать архитектуру программного продукта.
